1. Project overview

AutoLabel is a desktop application for annotating images and exporting YOLO segmentation datasets. The user opens an image folder, places positive and negative prompt points, and a SAM2 backend generates a mask; accepted masks become polygons and are exported in normalized YOLO segmentation format.

The frontend is C++ / Qt Widgets / OpenCV. The backend is a long-lived Python process (PyTorch + SAM2) that loads the model once and answers JSON requests over stdin/stdout, driven from Qt via QProcess. The project doubles as a structured learning path through modern C++, Qt, OpenCV, desktop app architecture, and C++/Python interop — and is intentionally kept small and focused (not a CVAT / Label Studio replacement).

InferenceVisualizer is the primary UI and architecture reference; SegmentAnythingAnnotator is the workflow / SAM-integration reference. Where the two disagree, AutoLabel follows InferenceVisualizer so the two apps feel like one software family.

2. Current phase Phase 3 / 10

Direction change (decided with the user). The original brief centred on SAM2 auto-masking via positive/negative point prompts. We pivoted: the manual annotation tool becomes the core (pure C++/Qt, no backend), and SAM2 is layered on later — eventually able to take a manually drawn shape and auto-predict polygons on subsequent images. Both modes will coexist. "First make it all work manually."

Phase 3 (this increment) is the manual annotation canvas. ImageView becomes the drawing surface: two modes — Rectangle (click two opposite corners → box, for detection) and Polygon (click vertices, click near the first to close, for segmentation) — with a live preview to the cursor. Esc aborts the in-progress shape, right-click undoes the last polygon vertex, Delete removes the last committed shape. On completion the shape is pending; Enter prompts for a class id and commits it. Committed shapes are stored in a new AnnotationModel and written to a YOLO .txt next to the image (detection format for rectangles, segmentation format for polygons, normalized); existing label files load back when an image is revisited.

Zoom/pan removed at the user's request — the image simply fits the window, and the canvas is dedicated to annotating. The Phase 2 View menu is gone.

Editing & colours (this increment). Committed shapes can now be edited, mode-agnostically: long-press a vertex to drag it, long-press an edge to insert a vertex and drag it, and right-click a shape for a context menu (Delete vertex / Delete shape). Hovering a vertex highlights it. Per-class colours arrived: each committed shape is drawn in its class's colour, configurable in a new Settings → Class Colours… dialog (a class-id → colour table backed by QColorDialog), with overrides persisted via QSettings; classes without an explicit colour fall back to a fixed default palette.

OpenCV decision (unchanged). The C++ frontend does not link OpenCV. A workspace scan found no C++ OpenCV installed (only the Python cv2 package). OpenCV-shaped work (mask→polygon extraction) will live in the Python SAM2 backend where cv2 already exists; the frontend stays on Qt for loading and display.

4. What was implemented

Phase 1

  • Two-tier CMake setup (root + src/) with Qt auto-detection that globs the newest C:\Qt\6.* install — identical to InferenceVisualizer so both projects configure the same way.
  • A top-level QMainWindow (src/app/MainWindow) with a title, status bar, a File menu (Open Image Folder… [Ctrl+O, stub] and Exit [Ctrl+Q]), and a Help menu (About).
  • A placeholder central widget (a centered QLabel) built in buildCentralWidget() — isolated behind one method so Phase 2 can swap in the image viewer with a small, local change.
  • Hand-coded UI (no .ui Designer files) so every action, shortcut, and connect() is visible and grep-able.
  • A file-based qInstallMessageHandler in main.cpp mirroring all qDebug/qWarning/qCritical output to autolabel.log next to the .exe — essential on Windows where Qt GUI apps route output through OutputDebugString.
  • Warnings-as-feedback compile flags (/W4 /permissive- on MSVC, -Wall -Wextra -Wpedantic otherwise) and a .vscode/launch.json debug profile.

Phase 2

  • New ImageView widget (src/ui/ImageView.{h,cpp}) built on QGraphicsView + QGraphicsScene + QGraphicsPixmapItem. Public API mirrors IV's ImageView (setImage / clear / hasImage).
  • Mouse-wheel zoom anchored under the cursor (AnchorUnderMouse), clamped to a sane scale range; click-drag pan via ScrollHandDrag; fit-to-window via fitInView. An m_autoFit flag keeps the image fitted on window resize until the user zooms manually, then honours their zoom level.
  • Empty-state hint ("No image loaded…") drawn directly on the viewport in paintEvent.
  • MainWindow: Open Image Folder… now scans the chosen directory (QDir::entryInfoList over a fixed extension set, sorted by name), loads the first image, and reports the count. Previous/Next navigation (←/→) under a new Image menu; Zoom In/Out (Ctrl+/-) and Fit to Window (Ctrl+0) under a new View menu. Navigation/zoom actions enable/disable based on state.
  • Images load via QImageReader with setAutoTransform(true) (EXIF orientation), per-file error reporting via QMessageBox. Status bar shows name [i/n] W×H; window title shows the current file name.

Phase 3 (manual annotation)

  • Direction change: manual annotation is now the core; SAM2 is deferred and will be layered on later. Both modes will coexist.
  • New annotation/ module: Shape (a rectangle [2 corner points] or polygon [≥3 vertices], in image pixels, plus a class id), AnnotationModel (the list of committed shapes for the current image + a changed() signal; pure data, no rendering/file knowledge), and YoloIo (free functions converting a shape list to/from a normalized YOLO .txt).
  • ImageView became the drawing canvas: Rectangle mode (two-click box) and Polygon mode (multi-click; click within ~12 px of the first vertex to close), each with a live cyan preview edge to the cursor. Drawing happens in drawForeground(), whose painter is already in scene == image-pixel coordinates; pens are cosmetic and vertex markers are sized as radius / scale so they stay constant on screen.
  • Interaction: Esc aborts the in-progress shape; right-click undoes the last polygon vertex; a completed shape is pending (drawn amber) until Enter, which prompts for a class id (QInputDialog, remembering the last value) and commits it; Delete removes the last committed shape.
  • MainWindow owns the model and is the only place that touches label files: on commit/remove it (re)writes <image>.txt next to the image; on opening an image it loads any existing .txt back so prior work is shown and editable. New Annotate menu toggles mode (R / P); the View menu and all zoom/pan were removed.

Phase 3 (editing + class colours)

  • Vertex editing of committed shapes, mode-agnostic and gesture-based: long-press a vertex (~350 ms hold) starts a drag; long-press an edge inserts a vertex at that point and drags it; moving more than a few px before the hold completes cancels the gesture so quick clicks still draw. A QTimer drives the hold; the dragged shape is edited on a working copy and written back to the model on release.
  • Deletion: right-click a committed shape opens a context menu (Delete vertex — polygons with >3 points only — / Delete shape); the existing Delete key still removes the last committed shape. Hovering a vertex highlights it with a white ring and an open-hand cursor.
  • AnnotationModel gained updateShape(i, …) and removeAt(i); the canvas emits a new annotationsChanged() after any committed-shape edit, which MainWindow connects to saveAnnotations() so the .txt stays in sync.
  • Per-class colours: new ClassColors (class id → colour; user overrides on top of a fixed default palette) and SettingsDialog (Settings → Class Colours…, Ctrl+,) — a class-id/colour table editable via QColorDialog, mirroring IV's swatch + QDialogButtonBox approach. Overrides persist via QSettings; committed shapes render in their class colour.

5. Why it was implemented this way

5.5 QGraphicsView for the image canvas, not a scaled QLabel

InferenceVisualizer's ImageView is a QLabel that rescales a pixmap to fit — perfect when you only ever "show the whole frame". AutoLabel needs more: Phase 2 requires real zoom + pan, and Phase 3 needs to convert a mouse click into an exact image-pixel coordinate for SAM2 prompts. QGraphicsView gives pan (ScrollHandDrag) and zoom (a scale() transform) almost for free, and because the pixmap item sits at the scene origin, mapToScene(pos) returns image pixel coordinates directly — exactly what Phase 3 wants. So the divergence from IV buys precisely the two capabilities this project needs.

// src/ui/ImageView.cpp — scene rect == image bounds, so scene coords == pixel coords
void ImageView::setImage(const QImage &image)
{
    resetInteraction();
    if (image.isNull()) { clear(); return; }
    m_pixmapItem->setPixmap(QPixmap::fromImage(image));
    m_scene->setSceneRect(m_pixmapItem->boundingRect());
    fitImage();
}

5.6 Zoom/pan removed — the canvas just fits the window

Phase 2 shipped mouse-wheel zoom and click-drag pan. The user asked to drop both: for this workflow the image should simply fill the window and the mouse is reserved for drawing. So ImageView sets NoDrag (left-click annotates, not pans), ignores the wheel, and re-fits the image on every resize. mapToScene() still yields exact image-pixel coordinates regardless of the fit scale, so annotation precision is unaffected. The View menu is gone.

5.7 No C++ OpenCV — OpenCV work goes to the Python backend

The original plan listed OpenCV as a frontend dependency. A workspace scan found no C++ OpenCV installed (only Python cv2), and on reflection the frontend doesn't need it: loading and display are a solved problem in Qt, and the only genuinely OpenCV-shaped task — turning an accepted mask into a simplified polygon (Phase 8) — naturally belongs in the Python SAM2 backend, which already imports cv2. Keeping OpenCV out of the C++ build removes a heavy native dependency and a Windows DLL deployment step for zero loss of capability.

5.8 Separation: canvas draws, model stores, MainWindow exports

Three responsibilities are kept in three places. AnnotationModel is pure data (a shape list + a changed() signal) and knows nothing about pixels-on-screen or files. ImageView renders that model and owns only the transient interaction state (the shape being drawn, the one awaiting a class) — it never touches the filesystem and doesn't know what a "class" dialog looks like. When a shape is finished, the canvas emits shapeDrawn(Shape); MainWindow is the only component that asks for a class, mutates the model, and writes the YOLO file. This is IV's "widget emits intent, app dispatches" pattern, so the file format, the class-entry UI, and the drawing logic can each change without disturbing the others.

// src/ui/ImageView.cpp — canvas finishes a shape, emits intent, forgets it
case Qt::Key_Return:
case Qt::Key_Enter:
    if (m_pending && m_pendingShape.isComplete()) {
        const ::Shape shape = m_pendingShape;
        resetInteraction();
        viewport()->update();
        emit shapeDrawn(shape);   // MainWindow assigns class + exports
    }
    return;

5.9 "Pending then Enter": a finished shape isn't committed until labelled

Completing a rectangle or closing a polygon does not immediately add it to the model. The shape enters a pending state (drawn amber), and only Enter → class dialog commits it; Esc discards it. This matches how the user described the flow ("when done, press Enter, type the class"), gives a clear chance to abort a misclick, and keeps every committed shape guaranteed to carry a class id — so export never has to deal with unlabelled geometry.

5.10 Our Shape vs QFrame::Shape

A subtle C++ gotcha worth recording: ImageView derives from QGraphicsView → QFrame, and QFrame has its own Shape enum. Inside the class the unqualified name Shape therefore resolves to QFrame::Shape, not our annotation struct — which first surfaced as a baffling connect() failure (the signal's parameter had silently become QFrame::Shape). The fix is to write ::Shape (the global one) inside this class; the rest of the codebase, which doesn't inherit QFrame, keeps the clean Shape name.

5.11 Long-press to edit, so drawing and editing share one canvas without a mode switch

The user asked specifically for a long-press to grab a vertex. That gesture neatly resolves the core ambiguity of a single canvas that both draws and edits: a quick click on empty space (or a quick click while mid-polygon) means "draw"; a deliberate hold on an existing vertex/edge means "edit". So there's no separate "edit mode" to toggle — editing is always available, but only triggers on an intentional hold, which can't be confused with placing a point. Implementation: a press over a committed feature starts a single-shot QTimer; if it fires before the button is released or the cursor wanders, the drag begins; otherwise the press is discarded as a tap. The drag mutates a working copy and only writes back to the model on release, so a half-finished drag never corrupts stored geometry.

5.12 Class colours: overrides over a default palette, persisted with QSettings

Class ids are open-ended (0, 1, 2…), so a fixed "class 0 / class 1" colour pair like IV's wouldn't scale. Instead ClassColors always answers colorFor(id): an explicit user override if one exists, else a deterministic pick from a 12-colour palette (id % 12). The user only adds rows in the Settings dialog for classes whose colour they want to pin; everything else is auto-coloured and still distinct. Overrides persist via QSettings (org/app already set in main.cpp), exactly like IV persists its inference settings.

5.1 Mirror InferenceVisualizer's two-tier CMake and src/ layout

The root CMakeLists.txt answers "what is this project, and where is Qt?"; the inner src/CMakeLists.txt answers "what files compile into the binary?". Adopting IV's exact structure (and the app/ · ui/ split it grows into) means patterns, naming, and build behavior carry straight over — the whole point of the two apps being one family.

5.2 OpenCV found but not linked yet

The architecture calls for OpenCV, but Phase 1 has nothing that uses it. Requiring it now would block a clean build for no benefit. IV added dependencies exactly when a step first needed them (Multimedia in step 2, ONNX Runtime in step 3); AutoLabel does the same — the find_package(OpenCV) line sits commented in the root CMake, ready for Phase 2.

5.3 Placeholder central widget behind buildCentralWidget()

Phase 1 has no image to show, so the central widget is a single centered QLabel. Isolating it in one method mirrors how IV's single pane became a QSplitter in one localized edit: Phase 2 replaces the body of buildCentralWidget() with the image viewer, and nothing else in MainWindow has to move.

5.4 The Open-Folder action exists now as a stub

Wiring the menu entry and its slot in Phase 1 settles the menu structure early; Phase 2 only fills in the body. The stub reports honestly via the status bar rather than pretending to work.

6. Build & run instructions

6.1 Windows command line (MinGW + Ninja)

$env:PATH = "C:\Qt\Tools\mingw1310_64\bin;C:\Qt\Tools\Ninja;C:\Qt\Tools\CMake_64\bin;$env:PATH"

cmake -S C:\Projects\AutoLabel `
      -B C:\Projects\AutoLabel\build `
      -G Ninja `
      -DCMAKE_BUILD_TYPE=Debug

cmake --build C:\Projects\AutoLabel\build

C:\Projects\AutoLabel\build\bin\AutoLabel.exe

CMake auto-detects the newest Qt 6.x under C:\Qt. Override with -DCMAKE_PREFIX_PATH=C:\Qt\6.11.1\mingw_64 if the glob picks the wrong one.

6.2 Qt Creator

  1. Open File → Open File or Project… and pick C:\Projects\AutoLabel\CMakeLists.txt.
  2. Select the MinGW 64-bit kit, click Configure Project.
  3. Build (Ctrl+B), run (Ctrl+R).

6.3 VS Code

The repo ships .vscode/launch.json (MinGW GDB on Windows, system GDB on Linux). With the CMake Tools and C/C++ extensions installed, pick the kit, then press F5 — the CMake: build preLaunchTask runs first.

7. Roadmap

  1. Phase 1 — Project skeleton. CMake, MainWindow, menus, status bar, placeholder.
  2. Phase 2 — Image viewer. Open folder, display/navigate images (Qt QImageReader → QGraphicsView). (Zoom/pan since removed in Phase 3.)
  3. Phase 3 — Manual annotation. (current) Rectangle (2-click box) + Polygon (multi-click, close near first) drawing, shape model, abort/undo/delete, class entry on Enter, YOLO .txt export + load next to each image.
  4. Phase 3-edit — Vertex editing + class colours. (done) Long-press a vertex to drag it; long-press an edge to add a vertex; right-click to delete vertex/shape; per-class colours via a Settings dialog.
  5. Phase 4 — Python backend comms. Long-lived QProcess, JSON request/response (ping → ok).
  6. Phase 5 — SAM2 service. Load model once; image + points in, mask out.
  7. Phase 6 — Mask visualization. Overlay with adjustable transparency, auto-refresh.
  8. Phase 7 — SAM2-assisted workflow. Positive/negative prompt points; Generate / Accept / Clear; accepted mask → polygon (OpenCV contour extraction in the Python backend). Later: seed SAM2 from a manual shape and auto-predict on the next images.

Roadmap reordered after the Phase 3 direction change: the manual tool (originally Phases 3, 8–10) is now the core and largely lands in Phase 3; SAM2 (originally Phases 4–7) is layered on afterward.
